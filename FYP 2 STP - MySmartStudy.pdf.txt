1.0 TEST POLICY  
This section outlines the testing philosophy and policies used in MySmartStudy, a full -stack 
Learning Management System developed for students and lecturers at IPGM Kampus 
Perempuan Melayu Melaka. The testing policy emphasizes compliance with organized, 
systematic, and industry -standard best practices, guided by software quality principles such 
as ISTQB and ISO 9000. Although the project is developed in an academic setting, the testing 
process is designed to simulate a professional software development envi ronment. Strong 
emphasis is placed on quality assurance to ensure that the platform effectively supports 
teaching, learning, and student adaptation through reliable digital learning tools.  
Testing is a critical phase of the Software Development Life Cycle (SDLC), particularly in 
ensuring that MySmartStudy’s multi -platform architecture functions correctly and 
consistently. The policy is intended to verify that the integrated architecture cons isting of the 
Next.js web frontend, Flutter mobile frontend, FastAPI backend, Firebase Authentication, and 
Firebase Firestore database satisfies both functional and non -functional requirements. Testing 
also ensures proper synchronization and consistent dat a flow between the web and mobile 
platforms through shared API services and Firebase -based authentication.  
The main objective of testing is to validate the functionality, integration, and reliability of the 
system’s major modules, including the mind map editor and viewer, course management, 
assignments and quizzes, discussion forum and class chat, attendance an d certificates, private 
messaging, notifications, planner/reminders, analytics, and AI -powered features such as the 
AI Study Companion, AI Timetable Analysis, AI Exam Planner, AI Study Materials, AI Plagiarism 
Detection, and AI Grading Assistant. Defects w ithin the web, mobile, backend, and cloud -
based components are identified early, documented carefully, and corrected through 
systematic testing to ensure that MySmartStudy remains dependable, secure, and effective as 
a digital learning platform.

1.1 Software Testing Term Definition  
Software testing is the systematic process of evaluating the MySmartStudy platform to identify 
defects, verify functionality, and ensure that the system operates according to its specified 
requirements. Testing involves examining and executing the major co mponents of the system, 
including the Next.js web application, Flutter mobile application, FastAPI backend, Firebase 
Authentication, Firebase Firestore database, and AI -supported modules, with the objective of 
detecting errors and confirming that actual re sults are consistent with expected results. In 
addition, testing ensures that shared services, cross -platform data flow, and user interactions 
function correctly across both web and mobile environments. This process is essential to 
guarantee that MySmartSt udy delivers reliable, secure, and effective support for teaching and 
learning activities Testing encompasses both functional such as  creating maps, saving notes, 
class management, analytics visibility and non -functional aspects like performance, usability,  
security, and cross -platform responsiveness. This process confirms the overall quality and 
reliability of the MySmartStudy application, which is crucial for supporting students and 
lecturers in digital learning environments.  
According to ISO -9000 principles, software testing is composed of two major components:  
• Validation  - Confirmation  by inspection and by the furnishing of objective. testimonies 
that the given requirements have been met. In simple terms, validation makes sure 
that the developed system is fit to the actual requirements of the user . 
 
• Verification - Confirmation  by observation and provision of objective. statements that 
required conditions to be satisfied to a specific application have been met. 
implemented. This would involve looking into the fact that the system was constructed 
in the proper way as per the. tec hnical specifications . 
During testing, the defects will be reported and logged down during testing including the 
wrong map rendering, authentication failures, synchronization, and inconsistency in the UI.

Testing is essential to ensure the quality, functionality, reliability, and usability of the 
MySmartStudy digital learning platform. Since the system is developed as a full -stack 
learning management system with a Next.js web frontend, Flutter mobile fronte nd, 
FastAPI backend, Firebase Authentication, and Firestore database, comprehensive testing 
is necessary to ensure smooth performance and a consistent user experience across 
multiple devices and platforms.  
The platform supports different user roles, mainly students, lecturers, and administrators, 
each of whom depends on core system functions such as mind map creation, course 
management, assignments, quizzes, planner/reminders, analytics, messaging, and AI -
powered learning support. Testing helps confirm that these features work correctly both 
as individual modules and as part of an integrated system. Through continuous and 
systematic testing during development, defects can be identified early before they affec t 
actual users.  Examples of defects that testing can help detect include : 
• Mind Map Feature Failures:  the mind map editor or viewer failing to save, load, 
render, or export maps properly across web and mobile environments.

1.3 Testing Principles  
The MySmartStudy project applies the seven core software testing principles outlined by 
ISTQB (International Software Testing Qualifications Board). These principles ensure that 
testing is carried out in a systematic, efficient, and practical manner for a full-stack digital 
learning platform that serves students, lecturers, and administrators. In MySmartStudy, 
these principles guide the testing of major system components such as the web frontend, 
mobile frontend, backend API, Firebase services, mind map fea tures, course tools, 
messaging, notifications, analytics, and AI -powered modules throughout the software 
development process.  
Principle 1: Testing shows the presence of defects  
Testing helps uncover defects, but it cannot guarantee that the system is entirely error -free.  
 For example, during testing of the Thinking Map creation module, issues may arise such as:  
• Maps not saving correctly  
• Wrong map type being recommended  
• Real -time collaboration failing to update changes  
These defects confirm that bugs exist even when other features appear to function well.  
Principle 2: Exhaustive testing is impossible  
It is not possible to test every input, action, device, user role, and usage scenario in 
MySmartStudy. Users may access the platform through different environments, create 
different types of learning content, submit assignments in multiple forms, interact through 
discussions and messaging, and use various AI -supported tools. Because of this, a risk-based 
testing approach  should be used, with priority given to high -impact and high -risk modules 
such as  
• Firebase Authentication and role -based access control   
• Cross -platform data consistency between web and mobile   
• Assignments, quizzes, and grading workflows   
• Notifications, messaging, and planner/reminders

Principle 3: Early testing  
Testing should begin as early as possible in the software development cycle.  
In the MySmartStudy project, test cases will be designed during early modeling and 
requirement stages of the Waterfall methodology.  
For example, while designing the collaboration feature, we can already create tests for:  
• Map sharing permissions  
• Real -time update accuracy  
• Access rights for students vs. lecturers  
Detecting issues early helps reduce rework, cost, and schedule delays.  
Principle 4: Defect clustering  
A majority of defects often reside in a small number of modules.  For MySmartStudy, more 
complex and sensitive modules may include:  
• Authentication and user synchronization  
• Mind map creation, collaboration, and export features  
• Assignments, quizzes, and gradebook calculations  
• Messaging, notifications, and forum discussions  
• Analytics dashboards and AI -supported modules  
Because these modules involve more complex logic, frequent data updates, or integration 
across multiple services, they require closer attention and repeated testing .

Principle 5: Pesticide paradox  
Running the same test cases repeatedly will eventually fail to uncover new defects.  
To overcome this, test cases must be regularly updated, especially when new features or 
enhancements are introduced.  For example:  
If new AI study tools, dashboard filters, or planner functions are added, new test scenarios 
should be created to verify correctness, usability, and performance. Updating test cases 
regularly helps maintain the effectiveness of the testing process : 
• Filter accuracy  
• Sorting behaviour  
• Data loading performance  
Refreshing test scenarios helps maintain defect discovery effectiveness.  
Principle 6: Testing is context dependent  
The testing approach must match the context, purpose, and user environment of the system. 
MySmartStudy is an academic digital -learning platform used by:  
• Students  (notetaking , Thinking Maps creation, collaboration)  
• Lecturers  (class monitoring, assignments, analytics)  
Thus, testing must reflect this context by focusing on usability, clarity, educational workflow 
accuracy, collaboration, and performance in a learning environment.  
Principle 7: Absence -of-errors fallacy  
If a system is free of defects but does not fulfil  user needs, it still fails.  Even if MySmartStudy 
has no technical bugs, it must still:  
• Recommend the correct Thinking Map types  
• Support smooth note organization

1.4 Test Missions & Objectives  
The mission of testing in the MySmartStudy  project is to ensure that the platform operates 
as a reliable, secure, and user -friendly digital learning system for students, lecturers, and 
administrators . Since the platform integrates a web application , mobile application , 
backend API , Firebase services , and AI-supported features , testing is necessary to confirm 
that each component functions correctly both independently and as part of the overall 
system. The purpose of testing is to ensure that the platform delivers accurate results, 
consistent cross -platform behaviour, proper rol e-based access, and a smooth user 
experience in supporting teaching and learning activities . 
Specific Objectives of Testing:  
• Ensure role -based access functions correctly:  Verify that students, lecturers, and 
administrators can log in successfully and access only the features and pages 
authorized for their roles. This includes confirming that restricted actions such as user 
management, grading, course administration, and ho mepage editing are protected 
from unauthorized access.  
• Validate mind map creation, editing, sharing, and saving:  Confirm that users can 
create, modify, organize, save, and view mind maps correctly, and that map -related 
data is stored and retrieved accurately through the system. Testing should also verify 
collaboration, annotations, and export -related functions where  applicable.

3.3.2 Test Design Techniques  
The following test design techniques will be used:  
• Equivalence Class Partitioning  
Divide input data into valid and invalid groups so that testing can be performed 
efficiently without repeating similar test cases. This technique will be applied to 
inputs such as login credentials, course fields, mind map titles, assignment forms, 
quiz da ta, and messaging inputs to confirm that the system accepts correct data 
while rejecting incorrect or incomplete entries . 
 
• Boundary Value Analysis  
To test the limits of input fields and numeric values in the system. This includes 
evaluating the maximum and minimum length of text fields such as titles, 
descriptions, and messages, as well as checking numerical boundaries for features 
such as grades, qu iz scores, and planner priorities . 
 
• Use Case Testing  
To ensure that the system behaves correctly during actual usage. This includes 
scenarios such as creating a mind map, joining a course, submitting an assignment, 
taking a quiz, sending messages, receiving notifications, and lecturers reviewing 
student work .

3.8 Risks and Contingencies  
Several potential risks could impact the testing process. Recognizing these risks allows the 
team to plan contingencies:  
• Ambiguous or Changing Requirements: If the project requirements are unclear or 
undergo late changes, test cases may become invalid. Contingency : Maintain close 
communication with the project supervisor and stakeholders. Freeze requirements 
before major testing activities and update test cases promptly when changes occur.  
• Limited Resources or Time Constraints: The project schedule may leave little time for 
thorough testing, or the test team may be understaffed. Contingency : Prioritize 
testing of critical features (risk -based) and use automated test tools where possible 
to save time. Allocate extra time for regression tests and consider extending 
deadlines if necessary.  
• Unavailability of Test Data or Environment: Required data sets or environments 
specific devices or browser versions may not be ready. Contingency : Create mock 
data or use virtualization simulators/emulators to mimic unavailable resources. Plan 
environment setup early and verify configurations in advance.  
• Technical Issues and Tool Reliability: Testing tools or the system under test may 
behave unexpectedly test scripts failing or system crashes . Contingency : Keep 
alternate tools or manual testing procedures as backups. Document any tool -related 
issues and seek technical support.  
• Personnel Risks: Key testers or developers may become unavailable illness, other 
assignments . Contingency : Cross -train team members on multiple features and 
maintain clear documentation so that others can pick up testing tasks if needed.

3.9 Test Levels (Black Box Testing, Risk -Based Testing)  
The testing strategy will incorporate multiple levels and approaches to ensure 
comprehensive coverage:  
• Unit/Component Testing: Individual components or functions of single page of the 
quiz module will be tested by developers. This white -box testing validates that each 
part of the code performs correctly in isolation. Defects found at this level are fixed 
before integration.  
• Integration Testing: After unit testing, related components for example, the mobile 
app, web application, backend API, authentication service, and database are tested 
together. This level ensures that data and control flow correctly between modules, 
verifying interfaces and da ta exchange. Integration tests will be partly white -box, 
with knowledge of the design, and partly black -box, focused on interactions.  
• System Testing Black Box Testing : The complete MySmartStudy system is tested from 
an end -user perspective without regard to internal code structure. Testers will 
execute functional test cases covering features such as mind maps, courses, 
assignments, quizzes, messaging, notifications, anal ytics, and AI -supported functions, 
verifying that the system meets the requirements. Black -box testing techniques such 
as equivalence partitioning and boundary value analysis will be used to design test 
cases. Usability an d UI consistency checks are also part of system testing . 
• Risk-Based Testing: Testing effort will be prioritized based on risk assessment. 
Features deemed most critical to the project’s objectives, for example 
authentication, role -based access, data synchronization, assignments, quizzes, 
analytics, and AI -supported modules, will be tested first and more thoroughly. This 
approach ensures that high -impact or high -probability -of-failure areas are covered 
early, allowing issues to be addressed promptly . 
• Regression Testing: Whenever changes are made such as bug fixes or minor updates, 
a regression test suite will be run to ensure existing functionality still works. This will

Testing for the MySmartStudy application will be conducted in a controlled test environment. 
The environment will simulate normal usage conditions to ensure consistent performance of 
features such as authentication, messaging, notifications, assignments, q uizzes, analytics, and 
data synchronization across web and mobile platforms.

• Android and iOS test devices (phones/tablets)  
• Laptops for web -based testing  
• Internet connection for Firebase Authentication , Firestore , and backend API access . 
• Access to testing servers, development tools, and Git . 
This environment allows testers to evaluate the application across multiple platforms and 
ensure seamless cross -device functionality.
